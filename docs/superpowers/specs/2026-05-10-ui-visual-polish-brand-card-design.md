# UI 视觉美化设计（Brand Card / 仅视觉层）

## 背景
当前 UI 已具备稳定交互能力（弹窗、toast、进度面板），但视觉层次与品牌识别度仍偏弱。此次仅做视觉升级，不调整任何交互逻辑、行为语义与调用协议。

## 目标
1. 统一 `dialog / progress / toast` 的视觉语言。
2. 在中性系统风基础上增加适度品牌色冲击。
3. 在不改 JS 逻辑的前提下提升可读性与质感。

## 非目标
1. 不修改任何交互行为（Esc、遮罩点击、按钮回调、toast 时长等保持原状）。
2. 不修改 `showDialog / showToast / showProgressPanel` 的 API 与参数。
3. 不引入第三方 UI 库或新增状态管理。

## 范围与边界
- 仅改 [src/shared/constants.js](src/shared/constants.js) 内 `BASE_STYLE`。
- 不改 [src/ui/dialog.js](src/ui/dialog.js)、[src/ui/toast.js](src/ui/toast.js)、[src/ui/progress.js](src/ui/progress.js) 的 DOM 结构与事件绑定。

## 方案对比与选择

### 方案 A：Balanced Brand Card（采用）
- 在现有结构上增强层次：更细腻的阴影、边框、圆角、间距与色彩分级。
- 主按钮强化品牌蓝，次按钮保持中性，状态色清晰但不过饱和。
- Toast 与 Progress 复用 Dialog 的视觉语义，形成统一体验。

### 方案 B：Bold Brand Emphasis
- 更高饱和主色与更强对比，视觉冲击更大。
- 风险是长期使用疲劳感偏高。

### 方案 C：Soft Premium Minimal
- 低饱和、弱对比、克制风格。
- 风险是品牌色存在感不足。

### 采用理由
方案 A 在“明显变好看”和“低风险可落地”之间平衡最佳，满足“稍有品牌色冲击”并避免过重视觉负担。

## 视觉设计

### 1) 全局基调
- 遮罩：蓝灰雾化感，保留内容可读性。
- 卡片：白底+细边框+分层阴影，提升浮层质感。
- 排版：通过字重和间距提升信息层级，不激进改字号。

### 2) Dialog
- 容器：圆角增大、阴影增强、留白更通透。
- 标题与正文：主次对比更明确，正文阅读更轻松。
- 按钮区：分割更清楚，主次按钮识别更直观。

### 3) 按钮体系
- 主按钮：品牌蓝为核心，hover 使用更深一阶蓝色。
- 次按钮：中性色维持低干扰。
- `focus-visible` 保持统一高可见 ring，避免可访问性回退。

### 4) Toast
- 与 Dialog 保持统一圆角、阴影、边框语义。
- 成功/失败色更纯净，信息态保持中性深色。
- 入退场动画保持轻量并更平滑（节奏不变）。

### 5) Progress
- 列表行增强轻卡片层级（微底色/边框/分隔）。
- 平台名与状态值层级更稳定，扫描效率更高。

## 架构与数据流影响
- 架构不变：仍由 `BASE_STYLE` 提供样式，UI 模块按原 class 渲染。
- 数据流不变：所有状态、回调、生命周期沿用现有逻辑。
- 错误处理不变：不新增异常路径，不改变关闭与回调时序。

## 兼容性与风险
1. 仅 CSS 文本替换，代码行为风险低。
2. 若未来调整 class 命名，本次样式需同步更新选择器。
3. userscript 注入环境下需注意过强阴影或高饱和色带来的视觉侵入，控制在“可感知但不刺眼”。

## 验证策略
1. 手动检查三组件是否视觉同源：弹窗、进度面板、toast。
2. 回归确认交互无变化：Esc、遮罩点击、按钮回调、toast 生命周期均与现状一致。
3. 运行现有测试集，确保无行为回归。

## 验收标准
1. 用户主观感受为“更现代、层次更好、品牌感更明显”。
2. 无任何交互行为变化与业务流程回退。
3. 代码改动仅位于 `BASE_STYLE` 对应样式定义区域。
